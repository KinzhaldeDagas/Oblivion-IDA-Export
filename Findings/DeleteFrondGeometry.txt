Deletion area map (where frond geometry is actually destroyed)

A) Primary explicit delete call site in build pipeline
- Function: BSTreeModel_MakeBase (sub_560600)
- Address: 0x5608D5 -> `call sub_789EC0`
- Sequence in success path:
  1) compute (`sub_78CCA0`)
  2) prep/update steps
  3) branch geometry delete (`sub_78C160`)
  4) frond geometry delete (`sub_789EC0`)  <-- main targeted deletion
- Practical meaning: tree/frond geometry is generated, consumed for setup, then frond payload can be dropped immediately to save memory.

B) The delete routine itself
- Function: sub_789EC0 (DeleteFrondGeometry)
- Core logic:
  - requires this+0x45 != 0 (computed state)
  - requires this+0x60 != 0 (frond geometry exists)
  - requires this+0x34 == 0
  - requires this+0x30 != 0 and *(*(this+0x30)) == 1 (ownership/refcount gate)
  - then:
    - `sub_797270(this+0x60_target)` deep-frees internals
    - `FormHeapFree(this+0x60)` frees outer frond geometry object
    - writes `this+0x60 = 0`

C) Destructor path also deletes frond geometry
- Function: sub_78C9B0 (global tree object teardown/destructor-like)
- If a1[0x18] exists:
  - calls `sub_797270(a1[0x18])`
  - calls `FormHeapFree(a1[0x18])`
  - later clears a1[0x18] to 0
- This means even if DeleteFrondGeometry is bypassed in MakeBase, final object teardown still eventually frees frond geometry.

D) Observable consumer behavior after delete
- Function: sub_78A390 (frond export/use path)
- If this+0x60 == 0, logs:
  - "no frond geometry exists, possible prior call to DeleteFrondGeometry"
- Confirms that nulling this+0x60 is the expected post-delete state and downstream code knows this case.

Ways to prevent deletion (ranked by safety and reversibility)

1) Preferred: skip the explicit call in BSTreeModel_MakeBase
- Patch location: 0x5608D5 (`call sub_789EC0`)
- Strategy: NOP out the call or redirect to a stub returning immediately.
- Effect:
  - preserves frond geometry after MakeBase success path
  - lowest behavioral blast radius (only this one callsite)
- Caveat:
  - frond memory remains until destructor path (sub_78C9B0), so memory footprint rises.

2) Soft block using DeleteFrondGeometry guard conditions
- Options:
  - force this+0x34 != 0 before delete call
  - force *(*(this+0x30)) != 1 before delete call
  - force this+0x45 = 0 before delete call (least desirable)
- Effect: function exits early; no free; this+0x60 remains valid.
- Caveat:
  - these flags may influence many other routines, so side effects can be wider than option #1.

3) Hook sub_789EC0 and early-return conditionally
- Runtime detour: intercept sub_789EC0 and return without running frees for selected tree classes/contexts.
- Effect: dynamic policy control (e.g., keep fronds only for debug captures).
- Caveat:
  - requires reliable context discrimination to avoid keeping too much memory alive.

4) Partial mitigation (NOT true prevention): intercept deep free only
- Hook sub_797270 to no-op while still allowing outer `FormHeapFree(this+0x60)`.
- Not recommended: this can leak internals and/or produce invalid state if outer object goes away.

5) Do not patch destructor path unless intentionally leaking
- Preventing delete in sub_78C9B0 would keep frond allocations alive past object teardown.
- High leak risk / lifetime correctness risk; only suitable for short-lived instrumentation runs.

Recommended implementation order
1) First patch only the MakeBase callsite (0x5608D5) and validate frond-dependent paths.
2) Monitor memory delta and lifetime behavior across scene transitions.
3) If needed, replace static patch with policy hook on sub_789EC0.
